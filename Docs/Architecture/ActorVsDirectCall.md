# 액터(Akka) 경유 vs 컨트롤러 직접 호출 — 장단점 정리

> 기록일: 2026-07-12
> 맥락: `Controller → Akka Actor → 유즈케이스 → DB` 처리 파이프라인을 랜딩 슬라이스에 도입한 뒤,
> "액터를 통해서 갈 때와 컨트롤러에서 직접 호출할 때의 장단점"을 정리한 학습 노트.

---

## 핵심 한 줄

액터를 넣는다는 건 **컨트롤러와 유즈케이스 사이에 "메일박스(단일 스레드 큐)"를 하나 끼우는 것**이다.
이 큐가 주는 성질이 **장점**이고, 그 큐를 건너느라 잃는 것이 **단점**이다.

```
직접 호출 :  Controller ── await ──▶ UseCase ─▶ Port ─▶ Repository ─▶ DB
액터 경유 :  Controller ─▶[ 메일박스 큐 ]─▶ Actor ─▶ UseCase ─▶ Port ─▶ Repository ─▶ DB
                          (비동기 메시지)      └ 메시지당 DI 스코프
```

---

## 액터 경유 — 장점

| 성질 | 무엇을 얻나 | 우리 프로젝트 예 |
|---|---|---|
| **동시성 제한(bulkhead)** | 풀 크기 N만큼만 동시에 DB를 친다. 요청 폭주 시 DB 커넥션 풀 고갈 방지 + **백프레셔** | 랜딩 풀 size 4 → DB 동시 조회 4개로 캡 |
| **엔티티별 직렬화** | ConsistentHash 라우터를 aggregate id로 키잡으면 **같은 엔티티 메시지가 한 액터로** 몰려 순차 처리 → DB 락 없이 경합 제거 | 팀 선수단 동시 수정, 경기결과 입력 |
| **무락(lock-free) 상태 보관** | 액터는 단일 스레드라 락 없이 in-memory 상태 유지 | KFA 동기화 rate-limit, 배치, 중복제거 카운터 |
| **버스트 흡수** | 메일박스가 자연스러운 큐 → 순간 부하 평탄화, 순서 보장 | 대회 결과 일괄 유입 |
| **결함 격리 / 감독(supervision)** | 실패한 액터만 재시작, 장애 전파 차단 | 외부연동 액터가 죽어도 조회는 생존 |
| **비동기 파이프라인 / 확장성** | fire-and-forget 알림, 스케줄링. 나중에 Akka.Cluster로 분산 시 **메시지 경계가 이미 존재** | 알림 발송, 지연 작업 |

## 액터 경유 — 단점

| 비용 | 내용 |
|---|---|
| **오버헤드·지연** | 메일박스 enqueue/dequeue + Ask마다 임시 액터·TaskCompletionSource·타임아웃 타이머 할당. 단순 읽기엔 **순수 낭비** |
| **⚠️ 요청 컨텍스트 소실** | `HttpContext`·요청 DI 스코프·`CancellationToken`·`ClaimsPrincipal`(로그인 사용자)·trace context가 **메일박스를 안 넘어감**. 필요한 값은 **메시지에 명시적으로 실어야** 함 |
| **수동 스코프 관리** | Scoped 유즈케이스/레포를 쓰려 메시지마다 `CreateScope()` 필요 (captive dependency 함정). 직접 호출은 요청 스코프를 공짜로 받음 |
| **실패 모드 증가** | Ask 타임아웃 / 메일박스 포화(dead-letter) / 액터 미등록 — HTTP로 매핑할 경우의 수 증가 |
| **디버깅 난이도** | 스택트레이스가 메일박스에서 끊김. 예외가 dead-letter/감독 이벤트로 바뀌어 추적이 어려움 |
| **규율 요구** | 메시지 불변, Sender는 await 전 캡처, 액터 안 블로킹 금지, 백프레셔 원하면 bounded mailbox 설정 필요 |

---

## 직접 호출은 정반대 성질

- **장점**: 최소 지연, 요청 컨텍스트(로그인 사용자·취소·스코프)가 자연히 흐름, 스택트레이스 온전, 실패 모드 적음, 코드 단순.
- **단점**: 동시성 제어·엔티티 직렬화·상태 코디네이션이 **전부 DB 몫**(락, 낙관적 동시성)이 됨. 버스트 완충 없음.

---

## 우리 프로젝트에서 특히 중요한 함정

**인증 엔드포인트(`api/player/me/...`)를 액터로 보낼 때** — 컨트롤러의 `User`(JWT 클레임)가 메일박스를 못 넘는다.
→ `userId`를 **요청 메시지에 넣어** 보내고, 액터 안에서 `HttpContext.User`를 참조하면 안 된다.
랜딩은 공개 조회라 이 문제가 없었지만, **본인 데이터 조회부터는 반드시 이 규칙**을 지켜야 한다.

---

## 판단 기준 (권장)

액터의 장점은 **쓰기 · 동시성 · 상태 · 외부연동**에서 나온다.

- **직접 호출로 충분** — 공개 읽기, 단순 CRUD 조회 (랜딩, 팀 검색, 경기기록 조회).
  액터를 껴도 얻는 게 거의 없고 지연만 늘어남.
- **액터가 값어치** — 엔티티별 순차성이 필요한 **쓰기**(선수단·경기결과·온보딩 저장),
  rate-limit 필요한 **외부연동**(KFA 동기화), 상태·배치.

즉 "모든 요청을 무조건 액터로"보다 **쓰기/코디네이션은 액터, 단순 조회는 직접 호출**로 나누는 게
비용 대비 효과가 크다.

다만 **하나의 경로로 통일**하는 것 자체의 가치(팀 규칙 단순화, 나중에 조회에도 캐시/스로틀 붙이기 쉬움)도
실재하므로, 그 트레이드오프는 팀 취향이다.
