# 공용 패턴 핸드오프 — 작업 순서 (2026-07-18 정리)

> ⚠️ **대체됨 (2026-07-18)** — 핸드오프 30종 기준의 `Handoff/PLAN.DEVELOPMENTORDER.md`가
> 이 문서를 **대체·확장**한다. **순서 판단은 그 문서 기준**으로 하고, 이 문서는 패턴 4부작 5종에
> 대한 초기 분석 기록으로만 남긴다(Phase 1·2·3 → 새 PLAN의 Phase A1~A4에 흡수).

새로 들어온 핸드오프 5종은 개별 화면이 아니라 **앱 전체에 적용되는 횡단 공용 패턴**이다.
각 화면 핸드오프에 비어 있던 "로딩·빈 상태·피드백·폼·연결" 자리를 이 5종이 채운다.
아래 순서는 **의존성 + 후속 기능 unblock**을 기준으로 정한 권장안이다(핵심 설계 결정 #3 YAGNI —
공용 컴포넌트로 한 번만 만들고 기존 화면에 소급 적용, 새 패턴 파생 금지가 5종 공통 규칙).

## 핸드오프 5종 요약

| 핸드오프 | 성격 | 산출물 | 규칙 핵심 |
|---|---|---|---|
| **Design.FormPatterns** | 입력/검증 (4부작) | 공용 필드 컴포넌트 + 검증 | 레이블 상단 고정(placeholder 대체 금지), blur 첫 검증→오류 시 즉시 재검증, 제출 시 첫 오류로 스크롤. **폼 오류=인라인(토스트 금지)** |
| **Design.FeedbackPatterns** | 토스트/모달 (4부작) | Toast 서비스(전역 1개 교체식) + ConfirmModal/BottomSheet | 완료=토스트, 파괴적 사전확인=모달(기본 포커스 취소), 오류 토스트 자동소멸 금지 |
| **Design.LoadingStates** | 로딩 스켈레톤 (4부작) | 시머 프리미티브 4종 + 인스턴스 6종 | 200ms 지연 표시, 실물과 레이아웃 동일(점프 0), 페이지 로딩은 스켈레톤(스피너는 버튼 제출만) |
| **Design.EmptyStates** | 빈 상태 카탈로그 (4부작) | Tier A(페이지)·Tier B(섹션) 공용 2종 + 인스턴스 12종 | CTA는 소유자에게만(방문자 안내 문구), 오렌지 금지, 자동 데이터는 입력 부담 제거 문구 |
| **Design.Navigation** | 연결 명세 | 라우트 접근 규칙 + GNB 상태 전환 + 에러 페이지(404/403/500) | returnUrl 보존, /dashboard 역할 분기(0/1/2+), 팀 공개홈 GNB 예외 |

## 권장 작업 순서

### Phase 1 — 입력 기반 (후속 기능 unblock) ★ 최우선

**1. Design.FormPatterns** — 공용 폼 필드/검증 컴포넌트
**2. Design.FeedbackPatterns** — Toast 서비스 + ConfirmModal/BottomSheet

> 근거: CLAUDE.md "다음 작업"의 미구현 기능(**경기 결과 입력 UI**, 팀 정보 수정, 커리어·포트폴리오
> 입력)이 전부 **폼**이다. 공용 폼/피드백을 먼저 세우면 그 기능들이 이 위에 올라가 divergence를 막는다.
> 둘은 짝(폼 오류=필드 인라인 / 저장 완료=토스트 / 삭제=확인 모달→실행취소 토스트)이라 함께 진행.
> 온보딩·로그인 등 **기존 폼도 이 컴포넌트로 소급 교체** 후 검수.

### Phase 2 — 로딩·빈 상태 (기존 화면 폴리시 + 일관화)

**3. Design.LoadingStates** — 스켈레톤 공용 컴포넌트 + 6 인스턴스
**4. Design.EmptyStates** — Tier A/B 공용 2종 + 12 인스턴스

> 근거: 이미 구현된 목록/카드 페이지(팀 대시보드·선수단·경기기록·공개홈·선수 대시보드)에 소급 적용.
> **주의**: 지금까지 만든 애드혹 점선 카드(커리어·포트폴리오·경기 결과·영상·시즌 통계·시즌성적 탭)는
> Tier A/B 카탈로그 문법으로 **통합**한다("새 패턴 만들지 말 것"). 둘은 한 쌍
> (로딩→스켈레톤→데이터 0건→빈 상태 흐름)이라 연달아. LoadingStates가 EmptyStates보다 선행.

### Phase 3 — 연결·에러 (전 화면 감사)

**5. Design.Navigation** — 라우트 접근 규칙 감사 + GNB 게스트/로그인 전환 + 에러 페이지

> 근거: 배선 대부분은 이미 있음(returnUrl·/dashboard 역할 분기·팀 공개홈 GNB 예외는 구현됨).
> **신규**는 에러 페이지 3종(404/403/500)과 **죽은 링크 0개 감사**(배선표의 모든 링크 실이동 확인).
> 전 화면이 어느 정도 구현된 지금이 감사 적기. GNB를 게스트↔로그인 전환 컴포넌트로 정리.
> `/teams`(팀 탐색), `/player/{id}`(공개 선수 프로필) 등 **미구현 화면 링크는 감사에서 제외**(구현 후 연결).

## 상호 의존 메모

- 4부작(Form·Feedback·Loading·Empty)은 서로 참조: 폼 오류는 토스트 아닌 인라인(Feedback 결정표),
  로딩 실패는 스켈레톤 자리를 에러로 교체(Navigation 500 축약), 데이터 0건은 빈 상태로.
- Navigation의 GNB/에러는 Feedback(토스트)·Empty(에러 안내)와 문법 공유 → Phase 1·2 이후 자연스러움.
- **미구현 선행 필요**: EmptyStates·Navigation의 일부 인스턴스는 아직 없는 화면(팀 탐색·공개 선수
  프로필·허브·알림 센터·설정·Claim 4스텝)을 참조 — 그 화면 구현 시 함께 배치, 지금은 **구현된 화면
  분량만** 적용.
