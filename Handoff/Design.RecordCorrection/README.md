# Handoff: 공식 기록 수정 신청 플로우 (Record Correction)

> 대상 저장소: `C:\Workspace\PlayGroundNew` · 압축 해제 위치: `Handoff\Design.RecordCorrection` (신규)
> 배경(7/19 확정): 공식 기록 = 주최측 소유. PlayGround는 **신청 제출 + 상태 확인 + 취소까지만** — 심사·반영 API는 별도 대회 서비스 몫.

## 규칙
- 신청 주체: 팀 관리자(팀 경기 전체) · 보호자(내 자녀 관련 기록만)
- **1건 1항목**(여러 오류 = 신청 여러 건 — 심사 단위 명확화) · 같은 경기에 내 미처리 신청 있으면 중복 신청 불가(⋯ 항목이 "신청 처리 중" 비활성으로)
- 상태 3종: **접수**(오렌지 톤, 취소 가능) → **반영**(teal) / **반려**(연레드, **사유 필수 표시**)
- 상태 변경(반영·반려) 시 알림 센터 발송 · 증빙 사진 첨부는 P1 보류

## 화면 명세
1. **진입점**: 공식 경기 행 ⋯ 메뉴(DropdownMenu) "기록 수정 신청" — **친선 행에는 없음**(직접 수정 가능)
2. **신청 폼** — PC 모달 / 모바일 바텀시트(FeedbackPatterns):
   - 헤더: 제목 + 경기 컨텍스트(대회·상대)
   - 오류 항목 라디오 카드 4종: 스코어 / 득점·도움 / 출전 선수 / 기타 (FormPatterns ≤4)
   - **현재 기록(자동 표시, 회색 읽기) → 올바른 기록(입력, 네이비 포커스)** 대비 구조 — 항목별 위젯: 스코어=숫자 2칸, 득점·도움/출전=선수 선택
   - 설명 "(선택)" · 성공 = 토스트 + 알림 센터 기록
3. **내 신청 목록**(TableList 정보형 행): 요약("스코어 3:1 → 3:2") + 신청일·주최측 + 상태 뱃지. 반려 행은 사유 박스 펼침. 위치: 팀 대시보드 기록 관리 절 + 보호자 허브 자녀 카드

## 데이터
- `CorrectionRequest`: MatchId · 항목 · 현재값 · 요청값 · 설명 · 상태(Pending/Accepted/Rejected) · 반려사유 · 신청자
- PlayGround API = 생성·조회·취소. 심사·반영은 대회 서비스(DB 공유)

## Files
- `Record Correction Flow.dc.html` — PC(진입 ⋯/모달 폼/상태 목록) / `Record Correction Flow Mobile.dc.html` — 모바일(바텀시트 폼/목록)
- `support.js` — 레퍼런스 실행용 (구현 대상 아님)

## Claude Code 첫 프롬프트 예시
```
Handoff/Design.RecordCorrection/README.md와 두 dc.html을 읽어.
1) CorrectionRequest 테이블 + 생성·조회·취소 SP/API (심사 API는 만들지 마 — 별도 서비스).
2) 공식 경기 행 ⋯에 "기록 수정 신청" 추가(친선 행 제외, 미처리 신청 시 비활성),
   신청 폼은 PC 모달/모바일 바텀시트 — 현재→올바른 기록 대비 구조, 1건 1항목.
3) 내 신청 목록(팀 대시보드 + 보호자 허브 자녀 카드), 상태 뱃지 3종 + 반려 사유,
   접수 상태 취소, 상태 변경 알림 센터 발송.
완료 후 신청→중복 차단→취소→반려 사유 표시 시나리오로 검수 요청해.
```
